в формате mermaid для следующего процесса:

Разработка внешнего расширения для баз 1С.
- разработчик расширения ведет разработку расширения в EDT используя конфигурацию с минимально необходимым количеством объектов метаданных и програмных модулей.
- разработка новой версии расширения ведется в ветке разработки с именем tXXXXXX где XXXXXX это номер ЗнИ в Интрасервис.
- по окончании разработки нужного функционала создается запрос на слияние в ветку main.
- после слияния в ветке main создается релиз с номером версии, который соответствует дате предполагаемого релиза.
- при создании релиза автоматически запускается процесс конвертации расширения в формат xml и помещение его в хранилище.
  - созданное хранилище расширения используется базами не подключенными к DevOps через КИП.
- при создании релиза автоматически запускается процесс публикации расширения в репозитории проектов которые на него подписаны.
  - "подписка на расширение" осуществляется с помощью создания в целевом репозитории ветки вида <Имя организации>_<Имя репозитория>_<Имя каталога> (далее ВеткаРасширения), где:
    - <Имя организации> это имя организации в которой расположен репозиторий с расширением.
    - <Имя репозитория> это имя репозитория с расширением.
    - <Имя каталога> это имя каталога в репозитории расширения идущее после точки.
  - после публикации расширения в репозитории проекта в ветку расширения, создается запрос на слияние в ветку main.
  - релиз менеджер принимает запрос на слияние и помещает таким образом обновленную версию расширения в каталог расширения в ветку main.
  - разработчики использующие функционал расширения должны обновить ветки в которых они ведут разработку до версии расширения в ветке main.
  - расширение используется для сборки конфигурации по стандартной схеме.
  - (Альтернативный вариант) расширение включается одновременно в файле проекта в исключаемые файлы (чтобы не собирать его при сборке проекта и не использовать дополнительное хранилище) и внешние (чтобы при обновлении тестовой и продуктивной базы добавлять туда расширение из единого хранилища). Плюсы варианта:
    - не требуется дополнительного хранилища расширения.
    - не требуется дополнительной конфигурации в проекте.
