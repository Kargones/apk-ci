Проанализируй файл требований @/promts/2-refact-03.md, включая секцию ## Вопросы для уточнения требований. Используй ответы на вопросы из этой секции как дополнительную информацию для уточнения и детализации требований. На основе полученных данных составь подробный документ SRS (Software Requirements Specification), структурируя требования по шаблонам EARS и группируя их по User Stories. Разделяй требования на следующие категории в соответствии с типами EARS: Ubiquitous, Event-driven, Unwanted Behavior, State-driven, Optional Features и Complex. Для каждой категории используй ключевые слова на русском языке ПРОПИСНЫМИ БУКВАМИ, за которыми следует шаблон и описание требования на русском языке, а также критерии приемки.

При формулировке требований соблюдай следующие правила:

- Присваивай каждому требованию уникальный идентификатор (например, REQ-001) для обеспечения traceability.
- Указывай четкие критерии приемки (например, тесты, метрики, условия выполнения).
- Избегай неоднозначных формулировок и терминов.
- При необходимости разбирай комплексные требования на атомарные элементы.
- Интегрируй архитектурные аспекты: опиши компоненты системы, интерфейсы, потоки данных и взаимодействия между частями системы.

Используй следующие шаблоны для структурирования требований:

```markdown
Ubiquitous (Повсеместные требования):

Шаблон: "The <system> shall <function>."
Описание: Требования, которые всегда активны и не зависят от событий или состояний.
Пример: "The authentication system shall verify user credentials within 2 seconds."
Когда использовать: Для базовых, постоянно действующих функций системы.

Event-driven (Событийные требования):

Шаблон: "When <event>, the <system> shall <function>."
Описание: Требования, активируемые определенным событием.
Пример: "When a user submits a form, the database shall store the data securely."
Когда использовать: Для реакции на внешние или внутренние события.

Unwanted Behavior (Требования к нежелательному поведению):

Шаблон: "If <condition>, then the <system> shall <function>."
Описание: Требования, описывающие обработку ошибок или исключительных ситуаций.
Пример: "If the network connection is lost, then the application shall retry the request up to 3 times."
Когда использовать: Для сценариев восстановления, обработки отказов и ошибок.

State-driven (Состоянийные требования):

Шаблон: "While <state>, the <system> shall <function>."
Описание: Требования, действующие только в определенном состоянии системы.
Пример: "While in manufacturing mode, the software shall boot without user intervention."
Когда использовать: Для систем с различными режимами работы.

Optional Features (Опциональные требования):

Шаблон: "Where <feature>, the <system> shall <function>."
Описание: Требования, зависящие от наличия или активации определенной функции.
Пример: "Where high availability is enabled, the system shall replicate data across nodes."
Когда использовать: Для конфигурируемых или дополнительных функций.

Complex (Комплексные требования):

Шаблон: Комбинация вышеуказанных шаблонов (например, "While <state>, if <condition>, then when <event>, the <system> shall <function>.").
Описание: Для сложных сценариев, где сочетаются несколько условий и событий.
Пример: "While in emergency mode, if battery is low, then when alert is triggered, the system shall send notification."
Когда использовать: В случае необходимости описания многоуровневых условий, но при этом стремись к декомпозиции на более простые требования.
```

Выделяй ключевые слова шаблонов ПРОПИСНЫМИ БУКВАМИ.

Генерируй SRS как полноценный документ, включающий следующие разделы:

1. Введение
2. User Stories
3. Требования по категориям EARS
4. Архитектурные аспекты (описание компонентов, интерфейсов, потоков данных)
5. Критерии приемки для каждого требования
