 Нужно создать эпик для реализации следующего функционала из описания из файла promts/lib-add-01.md
 - реализовать команду которая будет:
   - сканировать репозитории на наличие веток соответствующих шаблону <Имя организации>_<Имя репозитория>_<Имя каталога>
   - если ветка найдена, то:
    - обновлять содержимое каталога расширения в ветке <Имя организации>_<Имя репозитория>_<Имя каталога>
    - потом создавать создать запрос на слияние ветки <Имя организации>_<Имя репозитория>_<Имя каталога> в ветку main.

Для поиска веток во всех репозиториях сервера gitea на котором запускается action нужно использовать API gitea.
Если есть вопросы создай файл с описанием вопроса и предложи варианты ответов.

Ответы:
> 1. Область сканирования: сканируем ВСЕ репозитории сервера Gitea к которым можем получить доступ с помощью токена доступа. Сервер gitea корморативный,
 и количество репозиториев не будет слишком большим.
 2. Конфликты: что делать если в целевом репозитории уже есть изменения в каталоге расширения?
  - текущий репозиторий является источником истины для расширения, поэтому если в целевом репозитории есть изменения в каталоге расширения, то нужно привести состояние каталога в полное соответствие с текущим репозиторием, создать запрос на слияние в ветку main.
3. Версионирование: как трекать какая версия расширения уже была опубликована?
 - action запускается только при создании релиза в текущем репозитории расширения, поэтому номер релиза будет номером версии расширения. В запрос на слияние нужно добавлять нужную информацию, включая номер версии расширения и комментарии к релизу.

 Как должна запускаться эта команда?

  - Вариант A: Как часть релизного процесса (при создании release/tag автоматически), запуск только автоматически чтобы не создавать лишних запросов на слияние.

Что должно быть в описании Pull Request?
  - Полное: Версия + Release notes + ссылка на релиз + changelog

  Дополнительное замечание относительно:Входные данные:
  - GITEA_TOKEN (доступ к API) - да
  - Текущий репозиторий (org/repo) - да
  - Release tag + Release notes - нет, данные нужно получать с помощью API gitea
  - Каталог расширения (после точки в имени репо) - нет, данные нужно получать с помощью функции AnalyzeProject, которая возвращает имя проекта (это каталог до точки в имени каталога расширения) и имя расширения (это каталог после точки в имени каталога расширения).
